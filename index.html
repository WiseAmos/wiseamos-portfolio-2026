02 — PROCESS

HOW I
BUILD THINGS.

01

STATEFUL · 60%

Real-time stateful systems — inputs and outputs coupled, feedback tight. The kind where you can't lie about the latency budget because the test fails.

  • drones, control loops, trading
  • events in, events out, no drama
  • logs you can grep at 3am
02

DETERMINISTIC · 25%

Deterministic, predictable loops — bounded, observable, never surprising. The boring backbone that lets the interesting parts stay interesting.

  • queues, batch jobs, pipelines
  • idempotent, retried, audited
  • tests that actually catch the bug
03

STRUCTURED · 10%

Structured, faceted decisions — many small surfaces, one coherent whole. The kind of code that gets read more than written.

  • domain models, schemas, types
  • typed boundaries, not linter religion
  • docs that match the code
04

EMERGENT · 5%

Exploration at scale — many actors, one emerging pattern. Reserved for the projects where the answer isn't known yet and the budget is to find it.

  • multi-agent systems, sims, RL
  • research code that ships anyway
  • honest eval, not vibes

03 — ABOUT

I LIKE
HARD PROBLEMS
WITH REAL RAMIFICATIONS.

I'm Amos, an engineer in Singapore. I spend my time on the kind of systems where the failure modes are interesting — things with milliseconds of budget, things that move, things that have to keep running while the world around them changes.

I've shipped autonomous drones, an HFT framework against BTC perpetuals, a consumer face-lock app used by tens of thousands, a shopping-protection browser extension, and a handful of smaller tools. I write mostly Python, TypeScript, and Rust, with the occasional Solidity contract.

Outside of code, I follow low-latency trading research, race the occasional drone through a Singapore carpark, and try to keep the build-test-ship loop as short as the work allows.
